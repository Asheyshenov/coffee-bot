## 1. Current State Analysis

### 1.1 StatusSyncer Implementation (`lib/jobs/status_syncer.rb`)

The StatusSyncer job is **already implemented** but **not started**:

```ruby
module CoffeeBot::Jobs
  class StatusSyncer
    def initialize(interval_seconds: 2)  # 2-second polling interval
    def start    # Starts Thread.new loop
    def stop     # Stops the loop
    def run_once # Single execution cycle
  end
end
```

**Key characteristics:**
- Polls every 2 seconds (configurable)
- Uses separate Thread for non-blocking execution
- Queries orders with `INVOICE_CREATED` status
- Excludes expired orders (older than ORDER_EXPIRE_MINUTES)
- Calls `OrderService.sync_payment_status(order)` for each order
- Has proper error handling per-order

## 8. Architecture Risks

### 8.1 Thread Safety ⚠️ Low Risk

- Ruby MRI GIL prevents true parallel execution
- Database operations use Sequel which is thread-safe
- Each order processing is independent

**Mitigation:** Already handled by Sequel's connection pool

### 8.2 Resource Usage ⚠️ Low Risk

- 2-second polling is frequent but lightweight
- Only queries orders with INVOICE_CREATED status
- Limited to 50 orders per cycle

**Mitigation:** Query is already optimized with limit

### 8.3 Graceful Shutdown ⚠️ Medium Risk

- Thread may be killed mid-operation on bot stop
- Could leave order in inconsistent state

**Mitigation:** Add signal handlers for clean shutdown

### 8.4 Duplicate Processing ⚠️ Low Risk

- Callback and polling could race
- Same order could be processed twice

**Mitigation:** OrderService.sync_status checks current status before updating
