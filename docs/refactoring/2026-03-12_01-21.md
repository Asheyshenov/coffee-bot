### 1.2 Signature Verification

**File:** [`lib/services/mwallet/signer.rb:58-73`](lib/services/mwallet/signer.rb:58)

```ruby
def verify_callback(params)
  return false if params['hash'].nil? || params['hash'].empty?
  provided_hash = params['hash']
  params_without_hash = params.reject { |k, _| k == 'hash' }
  json_string = JSON.generate(params_without_hash)
  expected_hash = sign(json_string)
  secure_compare(provided_hash, expected_hash)
end
```

**Algorithm:** HMAC-MD5 over JSON body (without hash field)
**Status:** ✅ Correctly implemented with constant-time comparison

## 2. Root Cause Analysis

### 2.1 The Problem

When payment callback arrives:
1. ✅ Signature is verified
2. ✅ Duplicate is checked
3. ✅ Order is found
4. ✅ Status transitions to PAID
5. ❌ **Baristas are NOT notified**

### 2.2 Why It Matters

Without barista notification:
- Baristas don't see new paid orders immediately
- Orders sit in queue unnoticed
- Relies on polling or manual refresh

### 2.3 Current Workaround

StatusSyncer polls every 2 seconds:
- Calls `sync_payment_status`
- But `sync_status` in MWallet::Service also doesn't notify!

So even polling doesn't notify baristas automatically.

**✅ Compliant:**
- `CallbackApp` - HTTP layer only
- `QRPay::Service` - Business logic
- `QRPay::Signer` - Cryptographic operations
- `CallbackLog` - Persistence layer

#### Open/Closed Principle (OCP)

**⚠️ Minor Issue:**
- `OrderService.process_payment_callback` hardcodes MWallet:

```ruby
def self.process_payment_callback(params, raw_body)
  mwallet = MWallet::Service.new
  mwallet.process_callback(params, raw_body)
end
```

**Recommendation:** Factory pattern or provider detection based on callback route.

#### Liskov Substitution Principle (LSP)

**✅ Compliant:** Error classes follow proper inheritance.

#### Interface Segregation Principle (ISP)

**✅ Compliant:** Services expose only necessary public methods.

#### Dependency Inversion Principle (DIP)

**⚠️ Minor Issue:**
- Services directly instantiate client classes.
