## Root Cause Analysis

### Issue 1: Provider Field Not Updated

**Location:** [`lib/services/mwallet/mwallet_service.rb:50`](lib/services/mwallet/mwallet_service.rb:50)

**Problem:**
- Order created with default `provider: 'ODENGI_QRPAY'` (from migration)
- MWallet service creates invoice but never updates `provider` field
- This causes confusion about which provider should handle callbacks

**Code:**
```ruby
# lib/services/mwallet/mwallet_service.rb:50
def create_invoice_for_order(order)
  # ...
  order.set_invoice_data(invoice_data)
  order.transition_to!(OrderStatus::INVOICE_CREATED)
  # ❌ Missing: order.update(provider: 'MWALLET')
end
```

### Issue 2: Callback URL Not Publicly Accessible

**Location:** [`config/boot.rb:76`](config/boot.rb:76)

**Problem:**
- `PUBLIC_BASE_URL` defaults to `http://localhost:9292`
- MWallet API (external service) cannot reach localhost
- Callbacks silently fail (no retry, no notification)

**Code:**
```ruby
# config/boot.rb:61,76
PUBLIC_BASE_URL = ENV.fetch('PUBLIC_BASE_URL', 'http://localhost:9292')
MWALLET_CALLBACK_URL = "#{PUBLIC_BASE_URL}/callbacks/mwallet/result"
# Result: http://localhost:9292/callbacks/mwallet/result
# MWallet cannot reach this URL!
```

**Evidence:**
- Callback server running on port 4567 (not 9292)
- No entries in callback_logs table
- No incoming requests in server logs

### Issue 3: StatusSyncer Not Running

**Location:** [`bin/bot:32`](bin/bot:32) vs [`bin/callback_app`](bin/callback_app)

**Problem:**
- StatusSyncer runs inside `bin/bot` process (Telegram bot)
- Callback server runs in separate `bin/callback_app` process
- During test, only callback_app was running
- StatusSyncer was NOT running as fallback

**Code:**
```ruby
# bin/bot:32-37
status_syncer = CoffeeBot::Jobs::StatusSyncer.new(interval_seconds: 2)
status_syncer.start  # Only runs in bot process!
```

**Note:** StatusSyncer would have caught the payment via polling if it was running.

### Issue 4: No Client Notification on Payment

**Location:** Missing implementation

**Problem:**
- `Notifier#notify_payment_received` method exists
- Never called anywhere in the codebase
- Client doesn't receive confirmation after payment

**Evidence:**
```bash
$ grep -r "notify_payment_received" lib/
# (no results - method exists but never called)
```

## Recommendations

### Immediate Fixes (Required for Production)

1. **Update Provider Field**
   ```ruby
   # In MWallet::Service#create_invoice_for_order
   order.update(provider: 'MWALLET')
   ```

2. **Configure Public URL**
   ```bash
   # .env
   PUBLIC_BASE_URL=https://your-public-domain.com
   # OR use ngrok for development
   ```

3. **Run Both Processes**
   - Option A: Run bot and callback_app together
   - Option B: Add StatusSyncer to callback_app
   - Option C: Use proper process manager (foreman, systemd)

4. **Add Client Notification**
   ```ruby
   # In callback processing after mark_paid!
   notifier.notify_payment_received(order)
   ```

### Architectural Improvements

1. **Unified Process Management**
   - Use Foreman or Docker Compose for process orchestration
   - Ensure StatusSyncer runs alongside callback receiver

2. **Health Checks**
   - Add callback URL reachability check
   - Alert if callbacks not received within expected time

3. **Provider Abstraction**
   - Create PaymentProvider interface
   - Centralize provider field management

## Conclusion

The payment was successful on the provider side, but the bot didn't receive confirmation due to:

1. **Callback URL not reachable** (localhost)
2. **StatusSyncer not running** (polling fallback)
3. **Provider field not updated** (data inconsistency)

**Primary fix:** Configure `PUBLIC_BASE_URL` to a publicly accessible URL and ensure both bot (with StatusSyncer) and callback_app are running.

**Secondary fix:** Update provider field when creating MWallet invoices.
