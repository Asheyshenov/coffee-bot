## 1. Current State Analysis

### 1.1 Entry Point Structure (bin/bot)

```
bin/bot
├── Load configuration (boot.rb)
├── Load models
├── Load services
├── Load router
├── Create Telegram::Bot::Client
└── Run bot with router (blocking)
```

**Current responsibilities:**
- Dependency loading
- Bot initialization
- Error handling (Interrupt, StandardError)

**Missing:**
- Background jobs lifecycle management
- Graceful shutdown coordination

### 1.2 StatusSyncer Structure (lib/jobs/status_syncer.rb)

```
StatusSyncer
├── initialize(interval_seconds: 2)
├── start() → spawns Thread
├── stop() → sets @running = false
└── run_once() → syncs orders
```

**Current issues:**
- Thread is spawned but never joined on shutdown
- No signal handling for graceful stop
- No coordination with main process lifecycle

## 2. SOLID / SRP Analysis

### 2.1 Single Responsibility Principle

| Component | Current Responsibility | SRP Compliance |
|-----------|----------------------|----------------|
| `bin/bot` | Entry point + dependency loading + bot execution | ⚠️ Violated - should also manage jobs lifecycle |
| `StatusSyncer` | Poll and sync order statuses | ✅ Compliant |

### 2.2 Open/Closed Principle

`bin/bot` is closed for extension - adding new background jobs requires modifying the entry point.

**Recommendation:** Consider a JobManager pattern for multiple background jobs.

### 2.3 Dependency Inversion

Currently `bin/bot` directly instantiates specific classes. For better testability:

```ruby
# Current (tight coupling)
bot = Telegram::Bot::Client.new(token)

# Better (dependency injection)
bot = bot_class.new(token)
```

### 3.1 Where StatusSyncer Belongs

`StatusSyncer` is in **Infrastructure Layer** (`lib/jobs/`) - ✅ Correct placement.

### 3.2 Orchestration Responsibility

**Question:** Who should orchestrate background jobs?

**Options:**

| Option | Pros | Cons |
|--------|------|------|
| A: bin/bot directly | Simple, explicit | Mixes concerns |
| B: JobManager class | SRP compliant, extensible | Over-engineering for 1 job |
| C: Bot::Router | Co-located with bot logic | Wrong layer (presentation) |

**Recommendation:** Option A for now (YAGNI), refactor to JobManager when adding 2nd+ job.

## 4. Implementation Plan

### 4.1 Changes to bin/bot

```ruby
#!/usr/bin/env ruby
# frozen_string_literal: true

# ... existing requires ...

require_relative '../lib/jobs/status_syncer'

# Start the bot
log_info('Starting Coffee Bot...', 
  environment: CoffeeBot::Config::RACK_ENV,
  database: CoffeeBot::Config::DATABASE_URL.split('/').last
)

# Initialize background jobs
syncer = CoffeeBot::Jobs::StatusSyncer.new(interval_seconds: 2)

# Graceful shutdown handler
shutdown_handler = lambda do
  log_info('Shutting down gracefully...')
  syncer.stop
  log_info('Shutdown complete')
end

# Register signal handlers
%w[INT TERM].each do |signal|
  Signal.trap(signal) do
    shutdown_handler.call
    exit(0)
  end
end

begin
  # Start background jobs
  syncer.start
  
  # Run bot (blocking)
  bot = Telegram::Bot::Client.new(CoffeeBot::Config::TELEGRAM_BOT_TOKEN)
  bot.run do |bot_instance|
    CoffeeBot::Bot::Router.new(bot_instance).run
  end
rescue Interrupt
  log_info('Bot stopped by user')
  syncer.stop
rescue StandardError => e
  log_error('Bot crashed', error: e.message, backtrace: e.backtrace.first(5))
  syncer.stop
  raise
ensure
  syncer.stop
end
```

### 4.2 Changes to StatusSyncer (optional enhancement)

Add thread tracking for cleaner shutdown:

```ruby
def start
  return if @running
  
  @running = true
  log_info('StatusSyncer started', interval: @interval_seconds)
  
  @thread = Thread.new do
    while @running
      # ... existing loop ...
    end
  end
end

def stop
  @running = false
  @thread&.join(5)  # Wait up to 5 seconds for thread to finish
  log_info('StatusSyncer stopped')
end
```

## 5. Risk Analysis

### 5.1 Multiple Instances Risk

**Scenario:** Two bot instances running simultaneously.

**Impact:**
- Duplicate API calls to MWallet
- Potential race conditions on order status update

**Current Mitigations:**
| Protection | Location | Status |
|------------|----------|--------|
| Row-level lock | `Order.for_update` in StatusSyncer | ✅ Present |
| Status check | Only syncs `INVOICE_CREATED` | ✅ Present |
| Callback dedup | `CallbackLog.duplicate?` | ✅ Present |
| Idempotent transition | `mark_paid!` checks status | ✅ Present |

**Additional Protection (optional):**

```ruby
# Environment variable to disable syncer
DISABLE_STATUS_SYNCER = ENV.fetch('DISABLE_STATUS_SYNCER', 'false') == 'true'

syncer = CoffeeBot::Jobs::StatusSyncer.new(interval_seconds: 2)
syncer.start unless DISABLE_STATUS_SYNCER
```

### 5.2 Thread Safety

| Concern | Status |
|---------|--------|
| DB connection pool | ✅ Sequel is thread-safe |
| Shared state | ✅ @running is only written by main thread |
| Exception isolation | ✅ Caught in thread, logged |

### 5.3 Resource Leaks

| Concern | Mitigation |
|---------|------------|
| Thread not joined | Add `@thread&.join(5)` in stop() |
| DB connections | Sequel connection pool handles this |

## 6. Graceful Shutdown Flow

```
SIGINT/SIGTERM received
        ↓
Signal handler triggered
        ↓
syncer.stop() called
        ↓
@running = false
        ↓
Thread finishes current iteration
        ↓
Thread exits loop
        ↓
Thread.join(5) waits
        ↓
Main process exits
```

**Timeout handling:** If thread doesn't stop in 5 seconds, force exit.
